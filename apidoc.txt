--- Authentication ---
All authenticated endpoints require header: Authorization: Bearer <access_token>
Error 401: Missing or invalid JWT.

--- Request / Response ---
For POST/PATCH with JSON: Content-Type: application/json (text/plain also accepted).
For file uploads: Content-Type: multipart/form-data.
Success responses: { "success": true, "data": { ... } } or { "success": true, "message": "..." }
Paginated lists: ?page=1&page_size=20 (default page_size=20); response has count, next, previous, results.

--- Security ---
HTTPS, HSTS (production), rate limiting (DRF throttle + OTP rate limit).
JWT: Access token 15 min (RS256 when keys set), Refresh token 30 days (optional HTTP-only cookie).

1. AUTHENTICATION (/api/v1/auth/)
----------------------------------------
  Order of typical flows:
    - New signup: POST register/ → POST verify-otp/
    - Existing user login (mobile): POST register/mobile/ OR POST resend-otp/ (E164) → POST verify/mobile/ OR POST login/
  Error shape: Most views use { "success": false, "error": { "code": <http status>, "message": "..." } }.
  Validation errors (400) may use the global exception handler: error.message (summary) + error.details (field errors).

--- New registration flow (recommended) ---

POST /api/v1/auth/register/
  Auth: None
  Description: Start registration with name, phone, email, dob, gender. Stores pending data in cache until OTP is verified; no User row is created until POST verify-otp/ succeeds. matri_id is returned only after verify-otp/. Email and phone must be unique among verified users. If the same phone was used for an incomplete signup, pending data is replaced and a new OTP is sent.
  Rate limit: OTP rate limit applies (max 5 OTP requests per 10 minutes per identifier phone:{E164}; settings OTP_RATE_LIMIT_*).
  Body (JSON):
    {
      "name": "string (required)",
      "phone_number": "string (required, with country code, e.g. +919496954772)",
      "email": "string (optional, valid email)",
      "dob": "string (required, DD-MM-YYYY or DD/MM/YYYY, e.g. 16-12-1990)",
      "gender": "M | F | O (required)",
      "profile_for": "string (optional) — Who you are registering for: myself | son | daughter | brother | sister | friend | relative"
    }

  Validation rules:
    phone_number:
      - Required. Must include country code (e.g. +919496954772).
      - Validated with phonenumbers library; stored in E164 format.
      - Must be unique: if a user already exists with this phone and is_registered (mobile_verified=True), returns 400 "Phone number already registered".
      - If user exists with this phone but not verified, OTP is resent and profile updated (no duplicate).
    email:
      - Optional. If provided, validated with Django's validate_email; invalid format → "Invalid email address".
      - Must be unique among verified users: if another user has this email and mobile_verified=True, returns 400 "Email already registered".
      - If another unverified/pending registration holds this email on a different phone, returns 400 "Email already registered".
    dob:
      - Required. Allowed formats only: DD-MM-YYYY or DD/MM/YYYY (e.g. 16-12-1990 or 16/12/1990). Stored as date.
      - Calendar validation: invalid days (e.g. 31-02-2020) and non-leap 29 Feb rejected.
      - Must not be in the future ("DOB cannot be in the future").
      - Minimum realistic DOB: 1920-01-01 ("Date of birth is not realistic." for earlier dates).
      - Age: male (M) minimum 21 years; female (F) and other (O) minimum 18 years; maximum 80 years for all.
      - Example messages: "Invalid date format. Use DD-MM-YYYY or DD/MM/YYYY", "Minimum age for male is 21 years",
        "Minimum age for female is 18 years", "Minimum age is 18 years", "Maximum age allowed is 80 years",
        "Invalid date. Check day, month, and year (including leap years)."

  Success response 200:
    {
      "success": true,
      "message": "OTP has been sent to your phone number. Verify to create your account.",
      "data": {
        "phone_number": "+919496954772",
        "otp_sent": true,
        "registration_pending": true,
        "otp": "(only when DEBUG=true; omitted in production)"
      }
    }
    Note: matri_id is not returned here; it appears in POST verify-otp/ after successful verification.

  Error response 400:
    {
      "success": false,
      "error": {
        "code": 400,
        "message": "Phone number already registered"
      }
    }
    or "Email already registered", "Invalid email address", or other validation message in error.message.

  Error 429: Too many OTP requests (success: false, error: { code: 429, message }).

POST /api/v1/auth/resend-otp/
  Auth: None
  Description: Send a new 6-digit OTP for an E164 mobile (rate-limited). No 404: always sends OTP for valid
    phone when under rate limit. OTP key routing:
      - mobile_verified user -> mobile:{E164} (verify with POST verify/mobile/ or POST login/)
      - else -> phone:{E164} (verify with POST verify-otp/ after POST register/ has stored pending data)
    If there is no Redis pending registration yet, OTP still sends; user must call POST register/ to
    store profile data before verify-otp/ can create the account.
  Body (JSON):
    { "phone_number": "string (required; validated to E164, e.g. +919876543210)" }
  Rate limit: max 5 requests per 10 minutes per phone (separate Redis bucket resend_otp:{E164}; settings OTP_RATE_LIMIT_*).
  OTP: New 6-digit code; expiry from settings (default OTP_EXPIRY_MINUTES, e.g. 5 minutes).
  Response 200:
    {
      "success": true,
      "message": "OTP has been sent to your phone number.",
      "data": { "phone_number": "+91...", "otp_sent": true, "otp": "(only when DEBUG=true)" }
    }
  Error 429: { "success": false, "error": { "code": 429, "message": "Too many OTP requests. Try again after <n> minutes." } }
  Validation 400: Invalid/missing phone (phonenumbers validation) via exception handler.

---

POST /api/v1/auth/verify-otp/
  Auth: None
  Description: Verify OTP from POST register/ (identifier phone:{normalized_mobile}). Creates the User on first success
    when pending registration exists; otherwise activates existing user. Returns tokens and full profile payload.
  Body (JSON):
    {
      "phone_number": "string (required unless mobile is sent)",
      "mobile": "string (optional alias for phone_number)",
      "otp": "string (required, 6 digits)"
    }
  Note: Phone is normalized to E164 when possible (same as register). OTP must match the latest code for phone:{E164}.
  Response 200:
    {
      "success": true,
      "data": {
        "access_token": "JWT access token",
        "refresh_token": "JWT refresh token",
        "matri_id": "AM100001",
        "is_registered": true,
        "is_registration_profile_completed": false,
        "profile_status": "incomplete",
        "profile_steps": {
          "location": false,
          "religion": false,
          "personal": false,
          "family": false,
          "education": false,
          "about": false,
          "photos": false
        },
        "profile_completion_percentage": 0,
        "next_step": "location",
        "profile": {
          "id": "user pk",
          "matri_id": "AM100001",
          "basic_details": { "name", "gender", "dob", "email", "phone" },
          "photos": { "profile_photo", "full_photo", "selfie_photo", "family_photo", "aadhaar_front", "aadhaar_back" },
          "religion_details": { ... },
          "personal_details": { ... },
          "location_details": { ... },
          "family_details": { ... },
          "education_details": { ... },
          "about_me": "string"
        }
      }
    }
  Field explanations:
    access_token       — JWT for Authorization: Bearer <token>
    refresh_token      — JWT for token refresh (may also be in HTTP-only cookie)
    matri_id           — Unique profile ID (e.g. AM100001)
    is_registered      — Always true on successful verify
    is_registration_profile_completed — True when user has called POST /profile/complete/
    profile_status     — "completed" if all 7 steps done, else "incomplete"
    profile_steps      — Per-step completion: location, religion, personal, family, education, about, photos
    profile_completion_percentage — 0–100 (completed_steps / 7 * 100)
    next_step          — First incomplete step name, or null if all complete
    profile            — Full saved profile (same structure as GET /api/v1/profile/): basic_details, photos, religion_details, personal_details, location_details, family_details, education_details, about_me
  (Refresh token may also be set in HTTP-only cookie when JWT_REFRESH_COOKIE_NAME is set.)
  Error 400: Invalid/expired OTP, missing phone, or "Registration session expired or invalid..." (no pending registration and no user).
  Error 403: Account blocked — { "success": false, "error": { "code": 403, "message": "Your account has been blocked." } } (User.is_blocked=true).

--- Legacy / alternate auth ---

POST /api/v1/auth/register/mobile/
  Auth: None
  Use: Request login OTP for an existing mobile_verified user (raw mobile string as stored/submitted, max 20 chars; should match User.mobile).
  Body (JSON): { "mobile": "string (required, max 20 chars)" }
  Error 404: No verified account for that mobile — { "success": false, "error": { "code": 404, "message": "No registered account found with this mobile number. Please register first." } }
  Response 200:
    { "success": true, "message": "OTP sent to mobile", "data": { "mobile": "<mobile>", "otp": "(DEBUG=true only)" } }
  Note: No separate rate limit in code for this path; use resend-otp/ for E164 + dedicated resend rate limit.

POST /api/v1/auth/verify/mobile/
  Auth: None
  Description: Verify OTP sent via register/mobile/, resend-otp/, or register/ (tries identifiers mobile:{mobile} then phone:{mobile}).
  Body (JSON): { "mobile": "string", "otp": "string (6 digits)", "referral_code": "string (optional)" }
  Response 200: Same shape as POST verify-otp/ — { "success": true, "data": { access_token, refresh_token, matri_id, profile_*, profile, ... } }
    (via get_verify_otp_response_data). Refresh cookie set when JWT_REFRESH_COOKIE_NAME is configured.

POST /api/v1/auth/register/email/
  Auth: None
  Body (JSON): { "email": "string (required)" }
  Response 200: { "success": true, "message": "OTP sent to email", "data": { "email": "<email>" } }

POST /api/v1/auth/verify/email/
  Auth: None
  Body (JSON): { "email": "string", "otp": "string (6 digits)", "referral_code": "optional" }
  Response 200: { "success": true, "data": { "user": {...}, "tokens": { "access", "refresh" } } }

POST /api/v1/auth/login/
  Auth: None
  Description: Mobile + OTP login using identifier mobile:{mobile} (request OTP first via register/mobile/ or resend-otp/).
  Body (JSON): { "mobile": "string", "otp": "string (6 digits)" }
  Response 200: { "success": true, "data": { "user": { "id", "email", "mobile", "role" }, "tokens": { "access", "refresh" } } }

POST /api/v1/auth/token/refresh/
  Auth: None (refresh token in body or in HTTP-only cookie)
  Body (JSON): { "refresh": "string (optional if sent via cookie)" }
  Response 200: { "success": true, "data": { "tokens": { "access", "refresh" } } }

POST /api/v1/auth/logout/
  Auth: Required (JWT)
  Body (JSON): { "refresh": "string (optional; if provided and blacklist enabled, refresh token is blacklisted)" }
  Side effects: Sets user.tokens_invalid_before (invalidates existing access tokens); optionally blacklists submitted refresh; marks user offline (last_seen).
  Response 200: { "success": true, "message": "Logged out successfully." }

POST /api/v1/auth/password/reset/
  Auth: None
  Body (JSON): { "email": "optional", "mobile": "optional" } (at least one required)
  Response 200: If user found: { "success": true, "message": "OTP sent for password reset.", "data": { "email" or "mobile": "<sent-to>" } }
    If no user: { "success": true, "message": "If an account exists, you will receive OTP." } (same status 200; no enumeration).

POST /api/v1/auth/password/confirm/
  Auth: None
  Body (JSON): { "email" | "mobile", "otp", "new_password" (min 8 chars) }
  Response 200: { "success": true, "message": "Password updated." }

1a. ADMIN PANEL AUTHENTICATION (/api/v1/admin/auth/)
================================================================================
Admin panel login uses a separate AdminUser model (not matrimony member User).
Roles: admin | branch_manager | staff
Mobile input: 10 digits only (no +91 in request); stored as +91XXXXXXXXXX.

POST /api/v1/admin/auth/send-otp/
  Auth: None
  Description: Send OTP to AdminUser mobile for selected role.
  Body (JSON):
    {
      "role": "admin | branch_manager | staff (required)",
      "mobile": "string (required, 10 digits only, e.g. 9876543210)"
    }
  Validation / errors:
    - role invalid: 400 "Please select a valid role"
    - mobile invalid: 400 "Mobile number must be 10 digits"
    - mobile not found: 404 "No account found for this mobile number"
    - is_active=false: 403 "Your account has been deactivated. Contact admin."
    - rate limit: max 5 OTP requests per 10 minutes per mobile -> 429 "Too many OTP requests"
    - if locked after failed OTP attempts: 429 "Too many failed attempts"
  Response 200:
    { "success": true, "message": "OTP sent successfully", "data": { "mobile": "+919876543210" } }
    (In DEBUG=true, response may include "otp": "123456" for testing.)

POST /api/v1/admin/auth/verify-otp/
  Auth: None
  Description: Verify OTP and return JWT access + refresh tokens with role/branch/permissions.
  Body (JSON):
    {
      "role": "admin | branch_manager | staff (required)",
      "mobile": "string (required, 10 digits only)",
      "otp": "string (required, 6 digits)"
    }
  Rules:
    - OTP: 6 digits; expires in 10 minutes
    - Wrong OTP: 400 "Invalid OTP. Please try again." (max 3 attempts)
    - After 3 wrong attempts: lock for 30 minutes -> 429 "Too many failed attempts"
    - Expired OTP: 400 "OTP has expired. Please request a new one."
  Success response 200:
    {
      "success": true,
      "data": {
        "access_token": "JWT...",
        "refresh_token": "JWT...",
        "role": "admin",
        "name": "Anitha Lakshmi",
        "branch": { "id": 1, "name": "Chennai Central" },
        "permissions": ["manage_branches","manage_staff","view_reports"]
      }
    }

1b. ADMIN DASHBOARD APIs (/api/v1/admin/dashboard/)
================================================================================
All endpoints require Admin JWT authentication (from /api/v1/admin/auth/verify-otp/).
Header: Authorization: Bearer <access_token>

Overview & Staff Role Context
  The Staff Panel is a scoped subset of the Admin Panel and uses the same
  AdminUser authentication flow.
  Role scope rules:
    - Staff can only see data assigned to them.
    - Staff access same URL prefixes; scope is enforced server-side by JWT role.
    - Staff cannot approve commissions, mark salary paid, delete profiles, or export global reports.
    - Staff can create enquiries (auto-assigned) and add follow-up notes.
    - Staff can submit cash payment entries for customers (pending admin verification).

Standard success/error envelope:
  Success:
    { "success": true, "data": { ... } }
  Error:
    { "success": false, "error": { "code": 400, "message": "..." } }
  Paginated list:
    { "success": true, "data": { "count": N, "next": "...|null", "previous": "...|null", "results": [...] } }

Pagination defaults:
  - page_size default: 20
  - page_size max: 100
  - query params: ?page=1&page_size=20

1b-a. STAFF DASHBOARD APIs (/api/v1/staff/dashboard/)
================================================================================
Role: staff only. Uses Admin panel JWT (same as /api/v1/admin/auth/verify-otp/).
Header: Authorization: Bearer <access_token>

All endpoints: GET only. No query parameters. No request body.

Purpose: Staff Panel home ("Staff Dashboard") — KPI cards and recent activity scoped exclusively to the
logged-in staff member (no other staff data).

Error 401: Missing or invalid JWT — same as other Admin JWT endpoints (see top of file).

----------------------------------------------------------------------------
GET /api/v1/staff/dashboard/summary/
  Auth: Required (Admin JWT; role must be staff)
  Body: none

  Response 200:
    {
      "success": true,
      "data": {
        "staff_name": "Anitha Lakshmi",
        "branch": "Chennai Central",
        "my_profiles": {
          "count": 184,
          "growth": 6,
          "growth_pct": "+6"
        },
        "subscriptions_this_month": {
          "count": 12,
          "growth": 3,
          "growth_pct": "+3"
        },
        "commission_earned": {
          "amount": 14400,
          "growth_pct": "+18%"
        }
      }
    }

  data.staff_name — StaffProfile.name (page subtitle: "Welcome back, {staff_name}").
  data.branch — StaffProfile.branch.name (admin_panel Branch).
  data.my_profiles.count — total profiles assigned to this staff (CustomerStaffAssignment).
  data.my_profiles.growth — delta: (assignments created this calendar month) minus (assignments created previous calendar month).
  data.my_profiles.growth_pct — string label for that delta (e.g. "+6", "-2" if negative).
  data.subscriptions_this_month.count — successful plan_purchase transactions this month for users assigned to this staff.
  data.subscriptions_this_month.growth / growth_pct — month-over-month change in that count (same string convention as profiles).
  data.commission_earned.amount — integer (rupees): sum of commission_amt for status approved|paid with created_at in current month.
  data.commission_earned.growth_pct — vs previous month same sum, rounded percent with sign (e.g. "+18%"; if prior month 0 and current > 0, +100%; both 0 → "+0%").

  Error 403 — not staff role:
    { "success": false, "error": { "code": 403, "message": "Access denied. Staff token required." } }

  Error 403 — AdminUser.is_active is false, or StaffProfile exists but is_active is false:
    { "success": false, "error": { "code": 403, "message": "Your account has been deactivated. Contact admin." } }

  Error 400 — no StaffProfile for this user (admin_user link and legacy mobile match both fail):
    { "success": false, "error": { "code": 400, "message": "Staff profile not configured. Contact admin." } }

----------------------------------------------------------------------------
GET /api/v1/staff/dashboard/recent-activity/
  Auth: Required (Admin JWT; role must be staff)
  Body: none

  Response 200:
    {
      "success": true,
      "data": {
        "items": [
          {
            "id": 42,
            "timestamp": "25-03-2026 14:30",
            "actor_name": "Anitha Lakshmi",
            "actor_role": "staff",
            "action": "profile_verify",
            "action_display": "Profile Verified",
            "resource": "profile:AM100001",
            "details": "Profile verification set to True.",
            "ip_address": "192.168.1.1"
          }
        ]
      }
    }

  data.items — at most 5 AuditLog rows, newest first; actor = logged-in AdminUser only.
  timestamp — DD-MM-YYYY HH:MM (from serializer).
  action — machine code (e.g. profile_verify, commission_create); action_display — human-readable label.
  ip_address — may be null.

  Same error envelope and codes as GET /api/v1/staff/dashboard/summary/ (403 / 400 as above).

1c. ADMIN BRANCH MANAGEMENT APIs (/api/v1/admin/branches/)
================================================================================
All Branch Management endpoints require Admin JWT authentication.
Header: Authorization: Bearer <access_token>

Model: Branch
  - name: CharField(max_length=100, unique=True)
  - code: CharField(max_length=20, unique=True)  # auto-generated if not provided
  - city: CharField(max_length=100)
  - phone: CharField(max_length=20)
  - email: EmailField(unique=True)
  - address: TextField(blank=True)
  - is_active: BooleanField(default=True)
  - is_deleted: BooleanField(default=False, soft delete flag)
  - created_at: DateTimeField(auto_now_add=True)

Validation Rules
  - name:
      - required, min 3 chars, max 100
      - errors:
          - "Branch name is required"
          - "Name too short"
          - "A branch with this name already exists"
  - code:
      - optional in request; auto-generated if missing in POST
      - must be unique if provided
      - example format: "CHE-01", "CBE-01"
      - error: "Branch code already in use"
  - city:
      - required
      - error: "City is required"
  - phone:
      - required, valid Indian phone (digits only, min 10)
      - error: "Enter a valid phone number"
  - email:
      - required, valid format, unique
      - error: "Email already registered to another branch"

1d. ADMIN STAFF MANAGEMENT APIs (/api/v1/admin/staff/, /api/v1/branch/staff/)
================================================================================
Roles:
  - Admin: full CRUD for all branches
  - Branch Manager: can view/edit/create staff only in own branch
Header: Authorization: Bearer <access_token>

Model: StaffProfile (admin_panel.staff_mgmt)
  - emp_code: auto-generated "EMP001", "EMP002", ...
  - name, mobile, email, profile_photo
  - branch, designation, department, joining_date
  - basic_salary, commission_rate, monthly_target, achieved_target
  - pf_number, esi_number
  - street_address, city, state, pincode
  - bank_name, account_number, ifsc_code, upi_id
  - login_username, password (write-only; stored hashed)
  - is_active, is_deleted, created_at

Validation Rules
  - name: required -> "Staff name is required"
  - mobile: required, 10 digits, unique -> "Mobile already registered to another staff member"
  - email: optional, unique -> "Email already in use"

1d-b. ADMIN BULK UPLOAD APIs (/api/v1/admin/bulk-upload/)
================================================================================
Roles:
  - Admin: full access
  - Branch Manager: own data only
  - Staff: not allowed (403)
Header: Authorization: Bearer <access_token> (Admin JWT)

Accepted file types: .csv, .xlsx
Template columns (exact order):
  1) Name (required)
  2) Phone Number (optional, unique if provided)
  3) Email (optional)
  4) Date of Birth (optional, DD-MM-YYYY or DD/MM/YYYY)
  5) Gender (optional: M/F/O)
  6) Partner Preference (optional)
  7) Country (optional)
  8) State (optional)
  9) District (optional)
  10) City (optional)
  11) Address (optional)
  12) Religion (optional)
  13) Caste (optional; requires Religion)
  14) Mother Tongue (optional)
  15) Marital Status (optional)
  16) Has Children (optional: Yes/No)
  17) Number of Children (optional, integer >= 0)
  18) Height (cm) (optional, 100-250)
  19) Weight (kg) (optional, numeric)
  20) Complexion (optional: Fair/Wheatish/Dark/Very Fair)
  21) Highest Education (optional)
  22) Education Subject (optional)
  23) Employment (optional)
  24) Occupation (optional)
  25) Annual Income (optional)
  26) About Me (optional, max 500 chars)
  27) Family Type (optional)
  28) Father's Name (optional)
  29) Father's Occupation (optional)
  30) Mother's Name (optional)
  31) Mother's Occupation (optional)
  32) Family Status (optional)
  33) Number of Brothers (integer >= 0) (optional)
  34) Number of Married Brothers (integer >= 0) (optional)
  35) Number of Sisters (integer >= 0) (optional)
  36) Number of Married Sisters (integer >= 0) (optional)
  37) About My Family (optional)

Validation rules (errors format: { row, field, message }):
  - Name: required
  - Phone: optional; if provided must be 10 digits and unique (within file + DB)
  - Email: optional; if provided must be valid
  - DOB: optional; if provided must be DD-MM-YYYY or DD/MM/YYYY and not future
  - Gender: optional; if provided must be M/F/O
  - Caste requires Religion
  - Has Children required for divorced/widowed/separated
  - Number of Children required when Has Children=Yes
  - Height 100-250 if provided
  - Weight numeric if provided
  - Complexion one of Fair/Wheatish/Dark/Very Fair
  - About Me max 500 characters
  - Number of Brothers must be integer >= 0 if provided
  - Number of Married Brothers must be integer >= 0 if provided
  - Number of Sisters must be integer >= 0 if provided
  - Number of Married Sisters must be integer >= 0 if provided
  - Numeric family field error message: "Must be a non-negative integer." (row comes from errors[].row)
  - All new family fields are optional (no dependency rules)

Workflow:
  1) GET template
  2) POST validate (file) -> creates BulkUploadJob + returns job_id + validation_token
  3) POST import (validation_token required)
  4) <=50 rows import synchronously; >50 rows queued in Celery
  5) Poll status/{task_id} and list history/

----------------------------------------------------------------------------
GET /api/v1/admin/bulk-upload/template/

  Query: format=csv | xlsx (default: csv)
  Response 200: file download (Content-Disposition attachment)

    - XLSX: bulk_upload_template.xlsx

1d-c. ADMIN ENQUIRIES / LEAD PIPELINE (/api/v1/admin/enquiries/)
================================================================================
Header: Authorization: Bearer <access_token> (Admin JWT)

Role access:
  - Admin: full list/detail/kanban; create/move/note any; assign staff; filters branch_id, staff_id.
  - Branch Manager: enquiries for own branch only (matched via admin branch code); create/move/note/assign
    within branch; cannot create in another branch.
  - Staff: only enquiries assigned to them; create (defaults branch from staff profile if set, assigned_to=self);
    move/add note on own assignments only. Cannot assign staff.
  - Staff panel (recommended): scoped list/summary/detail/move/notes under GET/POST/PATCH /api/v1/staff/enquiries/
    (see section 1d-c-staff); behaviour matches staff rules above with staff-specific validation messages and 404 text.

Model: Enquiry
  - name (CharField), phone (CharField, 10 digits stored), email (optional)
  - source: website | walk-in | phone | whatsapp | email
  - status: new | contacted | interested | converted | lost (default new)
  - assigned_to → AdminUser (optional), branch → admin_panel.branches.Branch (optional)
  - notes (TextField, legacy single field on enquiry)
  - created_by → AdminUser (optional), created_at, updated_at

Model: EnquiryNote (timeline notes)
  - enquiry FK, text, created_by, created_at
  - Returned nested as enquiry_notes on detail/list serializers.

Query params (GET list):
  - search — name, phone, email (icontains)
  - status — new | contacted | interested | converted | lost
  - source — website | walk-in | phone | whatsapp | email
  - branch_id — admin only
  - staff_id — admin & branch manager (filter by assigned_to)
  - page, page_size — pagination (default page_size=20, max 100)

List response shape (DRF pagination, no success wrapper):
  { "count": N, "next": "url|null", "previous": "url|null", "results": [ ...Enquiry rows... ] }

Validation / error messages (400 unless noted):
  - Name is required.
  - Phone number is required. / Enter a valid 10-digit phone number.
  - Invalid source. Must be one of: website, walk-in, phone, whatsapp, email.

1f-b. BRANCH STAFF PERFORMANCE APIs (/api/v1/branch/staff-performance/)
================================================================================
Role: branch_manager only (Admin JWT). Staff role -> 403 "Insufficient permissions."
Auth: AdminJWTAuthentication.
Header: Authorization: Bearer <access_token>

Scope:
  - All metrics are for the branch manager's master.Branch (AdminUser.branch_id) and the matching
    admin_panel.branches.Branch (same code as /api/v1/branch/dashboard/).
  - Optional ?branch_id=<master_branch_pk> must equal the manager's branch; else 403 "Access denied".

Month window:
  - Optional ?month=YYYY-MM (calendar month). If omitted, the current calendar month is used.
  - Invalid format (e.g. 2026-13, abcd) -> 400:
    { "success": false, "error": { "code": 400, "message": "Invalid month format. Use YYYY-MM." } }

Definitions (same month window everywhere unless noted):
  - profiles_created: count of CustomerStaffAssignment rows for users in this branch where created_at falls
    in the month and the staff member belongs to this admin branch.
  - subscriptions_sold (per staff): count of successful plan_purchase Transaction rows for users assigned
    to that staff (CustomerStaffAssignment) in the month.
  - revenue (per staff): sum of Transaction.total_amount for those rows.
  - commission_earned: sum of Commission.commission_amt with status in (approved, paid), branch = manager's
    admin branch, created_at in the month (cancelled excluded).
  - conversion_rate: (subscriptions_sold / monthly_target) * 100, one decimal; monthly_target from StaffProfile;
    if monthly_target is 0, 100.0 when subscriptions_sold > 0 else 0.0.
  - target: StaffProfile.monthly_target; achieved: subscriptions_sold for that month.
  - status: Exceeded if achieved >= target; On Track if achieved >= target * 0.75; else Behind (if target <= 0,
    Exceeded when achieved > 0, else Behind).

KPI summary (GET .../summary/):
  - total_profiles_created: all new customer–staff assignments in the branch for the month (not only split by staff).
  - subscriptions_sold: all successful plan_purchase transactions for users in the branch in the month.
  - branch_revenue: sum of Transaction.total_amount for those branch subscription transactions.
  - avg_conversion_rate: simple average of per-staff conversion_rate for staff with monthly_target > 0, plus
    staff with target 0 and subscriptions > 0 counted as 100%; empty staff list -> 0.0.

Pagination (GET .../ root):
  - ?page=&page_size= (StandardPagination: default 20, max 100).

Export (GET .../export/):
  - Same query params as list (?month=&search=). Content-Type: text/csv; UTF-8 BOM for Excel.

  { "success": false, "error": { "code": N, "message": "..." } }

1g. ADMIN & BRANCH MANAGER COMMISSIONS LEDGER APIs (/api/v1/admin/commissions/, /api/v1/branch/commissions/, /api/v1/staff/commissions/)
================================================================================
Roles:
  - Admin: full control on /api/v1/admin/commissions/
  - Branch Manager: Commission Approval UI uses /api/v1/branch/commissions/ — KPI summary, list, detail, approve, cancel,
    bulk-approve, slip (own branch only); mark-paid is Admin-only (403 on branch mark-paid path)
  - Branch Manager: own branch read access on /api/v1/admin/commissions/ list/detail (POST manual create still allowed for own-branch staff)
  - Staff: My Commissions UI — dedicated read-only APIs under /api/v1/staff/commissions/ (admin_panel.staff_commissions):
    summary/, list, detail, export (no approve/cancel/pay; mutations return 403).
Header: Authorization: Bearer <access_token>

Model: Commission
  - staff (StaffProfile)
  - subscription (UserPlan)
  - customer (User)
  - branch (admin_panel.branches.Branch)
  - sale_amount
  - commission_rate
  - commission_amt
  - status: pending | approved | paid | cancelled
  - approved_by (AdminUser, nullable)
  - paid_at (nullable)
  - created_at

Auto-creation rule (see admin_panel.commissions.signals)
  - On each NEW successful Transaction with transaction_type=plan_purchase and payment_status=success,
    a Commission row may be created if ALL of the following hold:
    - The user has a UserPlan (subscription) — plan purchase flow creates UserPlan before recording the transaction.
    - A CustomerStaffAssignment exists for that user pointing to the staff who should earn commission
      (table admin_customer_staff_assignment; one staff per customer).
    - That StaffProfile is not deleted; staff.branch resolves (or user.branch matches a Branch by code).
  - Formula: commission_amt = sale_amount * commission_rate / 100, where sale_amount = transaction.total_amount
    and commission_rate is copied from the assigned staff’s commission_rate at that moment.
  - Set each employee’s default rate via admin staff APIs (staff.commission_rate).

Summary cards in list response:
  - total_pending
  - approved
  - paid
  - grand_total
  - (computed in a single DB aggregate for speed; list rows use select_related for staff/branch/customer/subscription/plan)

Validation / Business Rules
  - approve: only pending -> "Only pending commissions can be approved"
  - mark-paid: only approved -> "Commission must be approved before marking paid"
  - cancel: only pending -> "Only pending commissions can be cancelled"
  - bulk-approve: ids non-empty list -> "Please select at least one commission"
  - bulk-approve skips non-pending silently and returns approved_count
  - Staff cannot approve/mark-paid/cancel -> 403 "Insufficient permissions"
  - create (POST list): Admin or Branch Manager; Staff -> 403; branch manager limited to own-branch staff
  - Branch Manager dedicated endpoints (/api/v1/branch/commissions/): see subsection below for approve/cancel/mark-paid/slip/bulk rules and messages

1h. SALARY & PAYROLL APIs (/api/v1/admin/payroll/, /api/v1/branch/payroll/, /api/v1/staff/payroll/, /api/v1/staff/salary/)
================================================================================
Roles:
  - Admin: generate monthly salary, list all, summary KPIs, approve, mark paid, download slip (/api/v1/admin/payroll/)
  - Branch Manager: Branch Salary Management UI uses /api/v1/branch/payroll/ — KPI summary, list, detail, approve drafts,
    download slip (own branch); month query uses MM-YYYY; generate and mark-paid return 403 with explicit messages
  - Branch Manager (legacy): may still use admin payroll GET list/detail/download/approve with own-branch scope (YYYY-MM month)
  - Staff: My Salary UI — use /api/v1/staff/salary/ (admin_panel.staff_salary): YTD summary, current month preview,
    paginated history, PDF download (read-only). Legacy list: GET /api/v1/staff/payroll/ (also allows branch_manager;
    month-scoped admin-style list).
Header: Authorization: Bearer <access_token> (Admin JWT)

Model: SalaryRecord
  - staff (StaffProfile), branch (Branch), month (DateField, first day of month e.g. 01-02-2026)
  - basic, commission (sum of paid commissions in that month), allowances, deductions, gross, net
  - status: draft | approved | paid
  - approved_by (AdminUser, nullable), paid_at (nullable)
  - Unique (staff, month)

Commission column on generate:
  - Sum of admin_commission.commission_amt for this staff where status=paid and paid_at falls in that calendar month (timezone-aware bounds).

Validation / business rules
  - POST generate: month required DD-MM-YYYY (first day of month) — "Invalid month format, use DD-MM-YYYY (first day of month)"
  - POST generate: cannot target a month after the current month — "Cannot generate salary for future months"
  - POST generate: if any SalaryRecord exists for that month — 400 "Salary already generated for this month. Use individual edits."
  - PATCH approve: only draft — "Only draft records can be approved"; Branch Manager wrong branch — 403 "Access denied"
    (branch routes use "You can only approve salary records for your own branch staff.")
  - PATCH mark-paid: Admin only; only approved — "Salary must be approved before marking as paid"
  - POST generate: Admin only — 403 "Insufficient permissions"
  - Branch routes /api/v1/branch/payroll/: month param MM-YYYY — "Invalid month format. Use MM-YYYY."

     Body (JSON): { "name": "string", "gender": "male"|"female"|"other", "dob": "DD-MM-YYYY", "email": "string" }

     Body (JSON): { "country_id": int, "state_id": int, "district_id": int, "city_id": int, "address": string }

     Body (JSON): { "religion_id": int, "caste_id": int, "mother_tongue_id": int, "partner_religion_preference": string, "partner_preference_type", "partner_religion_ids", "partner_caste_preference" }

     height_cm stored as free text (e.g. "168 cm"). Response 200: { "success": true, "data": { ... } }

     Body (JSON): { "about_me": "string" }
     Response 200: { "success": true, "data": { "about_me": "..." } }

     Content-Type: multipart/form-data

POST /api/v1/profile/personal/
  Auth: Required (JWT)
  Body (JSON):
    {
      "marital_status": "string (optional, one of master_marital_status names e.g. Never Married, Separated, Widowed, Divorced; matched by name in master, case-insensitive)",
      "has_children": "boolean (optional, required when marital_status is Separated, Widowed or Divorced)",
      "number_of_children": "integer (optional, required when has_children is true, ≥ 0)",
      "height": "integer (optional, height in cm, e.g. 175)",
      "weight": "decimal (optional, e.g. 65.5)",
      "complexion": "string (optional, one of: Fair, Wheatish, Dark, Very Fair)"
    }
  Validation rules:
    marital_status:
      - Optional. When provided, must match an active record in master_marital_status by name (case-insensitive).
      - If marital_status is "Separated", "Widowed" or "Divorced", has_children becomes required.
    has_children:
      - Optional normally.
      - When marital_status is "Separated", "Widowed" or "Divorced" and has_children is missing or null → 400 with message "Please specify whether you have children."
    number_of_children:
      - Required when has_children is true.
      - Must be an integer ≥ 0.
    height:
      - Optional integer in centimeters; stored as height in cm for the profile (e.g. 175).
    weight:
      - Optional decimal value (e.g. 65.0, 65.5).
    complexion:
      - Optional.
      - Must be one of: "Fair", "Wheatish", "Dark", "Very Fair" (case-sensitive).

  Example valid request:
    {
      "marital_status": "Separated",
      "has_children": true,
      "number_of_children": 2,
      "height": 175,
      "weight": 65.5,
      "complexion": "Wheatish"
    }

  Example invalid request (missing has_children for Separated):
    {
      "marital_status": "Separated"
    }

  Example invalid response 400:
    {
      "success": false,
      "error": {
        "code": 400,
        "message": "Please specify whether you have children."
      }
    }

  Success response 200:
    {
      "success": true,
      "message": "Personal details saved successfully."
    }

  Error responses:
    400 → Validation error (e.g. invalid marital_status, missing has_children for Separated/Widowed/Divorced, missing number_of_children when has_children is true, invalid complexion value).
    401 → Unauthorized (missing or invalid JWT).

WebSocket — Real-time chat (Django Channels)
--------------------------------------------------------------------------------
  URL: ws://<host>/ws/chat/<conversation_id>/
  Authentication: JWT passed as query parameter: ?token=<access_token>
  Example: ws://127.0.0.1:8000/ws/chat/1/?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  The user must be a participant in the conversation; otherwise the connection is rejected (403).

  After connecting, send JSON text frames to post a message:
    { "message": "Hello!" }
  or
    { "text": "Hello!" }

  Incoming messages (broadcast to both participants) have the form:
    {
      "message_id": 101,
      "sender_id": "uuid",
      "sender_matri_id": "AM100001",
      "sender_name": "Anna",
      "text": "Hello!",
      "created_at": "13-03-2026"
    }

  Messages are persisted to the database and conversation.updated_at is updated on each new message.
  Backend uses Redis channel layer (channels_redis) for broadcasting between workers.

  Online status: Connecting to this WebSocket (or sending a message) updates the user's last_seen so
  the other participant sees is_online in GET /api/v1/chat/list/ and GET /api/v1/chat/messages/...
  when they refetch (within 15 minutes of activity).

  WebSocket JSON events (in addition to chat messages):
    - On connect, server sends the other user's current status:
      { "type": "presence", "matri_id": "AM100006", "online": true|false }
    - When the other participant connects to this chat WebSocket, you receive:
      { "type": "presence", "matri_id": "<their matri_id>", "online": true }
      Frontend should show "Online" when online is true for the chat partner's matri_id.

Only returns transactions belonging to the authenticated user.
Transactions ordered by created_at descending.

Transaction types: plan_purchase | profile_boost | refund
Transaction statuses: success | failed | pending | refunded
Payment methods: razorpay | stripe | upi | manual

--- API 1: Transaction Summary ---

GET /api/v1/transactions/summary/
  Auth: Required
  Description: Returns dashboard summary cards — total spent, active plan name, next renewal date.
  Response (200):
    {
      "success": true,
      "data": {
        "total_spent": 18995.0,
        "active_plan": "Gold Plan",
        "next_renewal": "01-06-2026"
      }
    }
  Notes:
    - total_spent: sum of all successful (payment_status=success) plan_purchase transactions.
    - active_plan: name of the user's currently active UserPlan (null if none).
    - next_renewal: UserPlan.valid_until formatted as DD-MM-YYYY (null if no active plan).

SUCCESS STORIES (Admin Panel)
================================================================================
Role access:
  - Admin: full CRUD, publish, toggle featured
  - Branch Manager: view only (list/detail)
  - Staff: view only (list/detail)

Entity fields:
  - couple_name_1 (string, required)        # Groom's Name
  - couple_name_2 (string, required)        # Bride's Name
  - wedding_date  (date, required, DD-MM-YYYY)
  - location      (string, required)
  - story_text    (string, required, min 50 chars)
  - couple_photo  (file, optional; JPEG/PNG/WEBP, max 5MB)
  - status        (draft|published)
  - is_featured   (boolean)
  - views_count   (integer, read-only)

Validation rules/messages:
  - If either couple_name_1 or couple_name_2 missing/blank:
      "Both partner names are required"
  - wedding_date in future:
      "Wedding date cannot be in the future"
  - location missing/blank:
      "Location is required"
  - story_text shorter than 50:
      "Story must be at least 50 characters"
  - Invalid photo type/size:
      "Photo must be JPEG/PNG/WEBP under 5MB"
  - Publish when already published:
      "Story is already published"
  - Only one featured story is allowed at a time.
    When a story is featured, previous featured story is automatically unfeatured.

--- UPDATE SUCCESS STORY ---
PATCH /api/v1/admin/success-stories/{id}/
  Auth: Required (Admin JWT)
  Content-Type: multipart/form-data or application/json
  Body: any editable fields
  Response 200:
    { "success": true, "data": { ...updated story... } }
  Error 400/403/404 as above.

--- DELETE SUCCESS STORY ---
DELETE /api/v1/admin/success-stories/{id}/
  Auth: Required (Admin JWT)
  Response 200:
    { "success": true, "message": "Story deleted successfully" }
  Error 403/404 as above.

--- PUBLISH SUCCESS STORY ---
PATCH /api/v1/admin/success-stories/{id}/publish/
  Auth: Required (Admin JWT)
  Logic:
    - Allowed only when current status is draft.
    - Sets status to published.
  Response 200:
    { "success": true, "data": { ...story with status: \"published\"... } }
  Error 400:
    { "success": false, "error": { "code": 400, "message": "Story is already published" } }
  Error 403/404 as above.

--- TOGGLE FEATURED ---
PATCH /api/v1/admin/success-stories/{id}/toggle-featured/
  Auth: Required (Admin JWT)
  Logic:
    - Toggles selected story is_featured true/false.
    - If toggled to true, all other featured stories are unset.
  Response 200:
    { "success": true, "data": { ...story with updated is_featured... } }
  Error 403/404 as above.

REPORTS & ANALYTICS (Admin Panel)
================================================================================
Base path: /api/v1/admin/reports/
Auth: Required (Admin JWT)

Role access:
  - Admin: all branches (optional ?branch_id= filter — master Branch.id, same as User.branch_id)
  - Branch Manager: own branch only (branch_id ignored; scoped automatically)
  - Staff: not allowed (403)

Common query params (where applicable):
  - branch_id — optional (Admin only); master Branch primary key
  - start_date, end_date — DD-MM-YYYY (optional; defaults vary by report)
  - start, end — aliases for start_date / end_date

Revenue basis: successful plan_purchase transactions only (payment_status=success, transaction_type=plan_purchase).

--- REVENUE REPORT ---
GET /api/v1/admin/reports/revenue/
  Query params:
    period — required style: daily | monthly | yearly (default monthly)
    start_date, end_date (or start, end) — optional; if omitted:
      daily: last 30 days
      monthly: last 365 days

    branch_id — optional (Admin only; master branch id)
