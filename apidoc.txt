================================================================================
MATRIMONY BACKEND - API DOCUMENTATION (Aiswarya Matrimony)
================================================================================
Base URL: /api/v1
Auth Base: /api/v1/auth/
Profile Base: /api/v1/profile/
Master Base: /api/v1/master/

--- Authentication ---
All authenticated endpoints require header: Authorization: Bearer <access_token>
Error 401: Missing or invalid JWT.

--- Request / Response ---
For POST/PATCH with JSON: Content-Type: application/json (text/plain also accepted).
For file uploads: Content-Type: multipart/form-data.
Success responses: { "success": true, "data": { ... } } or { "success": true, "message": "..." }
Paginated lists: ?page=1&page_size=20 (default page_size=20); response has count, next, previous, results.

--- Security ---
HTTPS, HSTS (production), rate limiting (DRF throttle + OTP rate limit).
JWT: Access token 15 min (RS256 when keys set), Refresh token 30 days (optional HTTP-only cookie).

--- New registration flow (recommended) ---

POST /api/v1/auth/register/
  Auth: None
  Description: Register new user with name, phone, email, dob, gender. Email and phone must be unique (among verified users). If user exists with same phone but not yet verified, resend OTP and update details (no duplicate user). Generates matri_id and sends OTP.
  Rate limit: OTP rate limit applies (max 5 OTP requests per 10 minutes per phone/email).
  Body (JSON):
    {
      "name": "string (required)",
      "phone_number": "string (required, with country code, e.g. +919496954772)",
      "email": "string (required, valid email)",
      "dob": "string (required, DD-MM-YYYY, e.g. 16-12-1990)",
      "gender": "M | F | O (required)"
    }

  Validation rules:
    phone_number:
      - Required. Must include country code (e.g. +919496954772).
      - Validated with phonenumbers library; stored in E164 format.
      - Must be unique: if a user already exists with this phone and is_registered (mobile_verified=True), returns 400 "Phone number already registered".
      - If user exists with this phone but not verified, OTP is resent and profile updated (no duplicate).
    email:
      - Required. Validated with Django's validate_email.
      - Must be unique: if a user already exists with this email and is_registered (mobile_verified=True), returns 400 "Email already registered".
      - Invalid format returns 400 "Invalid email address".
    dob:
      - Required. Format DD-MM-YYYY (e.g. 16-12-1990). Stored as date.
      - Must not be in the future. Invalid format returns 400.

  Success response 200:
    {
      "success": true,
      "message": "Registration successful. OTP has been sent to your phone number. Please verify to continue.",
      "data": {
        "matri_id": "AM100002",
        "phone_number": "+919496954772",
        "otp_sent": true
      }
    }

  Error response 400:
    {
      "success": false,
      "error": {
        "code": 400,
        "message": "Phone number already registered"
      }
    }
    or "Email already registered", "Invalid email address", or other validation message in error.message.

  Error 429: Too many OTP requests (success: false, error: { code: 429, message }).

POST /api/v1/auth/verify-otp/
  Auth: None
  Description: Verify OTP sent to phone. Returns tokens, multi-step profile completion status, and all previously saved profile data in detail.
  Body (JSON):
    {
      "phone_number": "string (required)",
      "otp": "string (required, 6 digits)"
    }
  Response 200:
    {
      "success": true,
      "data": {
        "access_token": "JWT access token",
        "refresh_token": "JWT refresh token",
        "matri_id": "AM100001",
        "is_registered": true,
        "is_registration_profile_completed": false,
        "profile_status": "incomplete",
        "profile_steps": {
          "location": false,
          "religion": false,
          "personal": false,
          "education": false,
          "about": false,
          "photos": false
        },
        "profile_completion_percentage": 0,
        "next_step": "location",
        "profile": {
          "id": "user pk",
          "matri_id": "AM100001",
          "basic_details": { "name", "gender", "dob", "email", "phone" },
          "photos": { "profile_photo", "full_photo", "selfie_photo", "family_photo", "aadhaar_front", "aadhaar_back" },
          "religion_details": { ... },
          "personal_details": { ... },
          "location_details": { ... },
          "family_details": { ... },
          "education_details": { ... },
          "about_me": "string"
        }
      }
    }
  Field explanations:
    access_token       — JWT for Authorization: Bearer <token>
    refresh_token      — JWT for token refresh (may also be in HTTP-only cookie)
    matri_id           — Unique profile ID (e.g. AM100001)
    is_registered      — Always true on successful verify
    is_registration_profile_completed — True when user has called POST /profile/complete/
    profile_status     — "completed" if all 6 steps done, else "incomplete"
    profile_steps      — Per-step completion: location, religion, personal, education, about, photos
    profile_completion_percentage — 0–100 (completed_steps / 6 * 100)
    next_step          — First incomplete step name, or null if all complete
    profile            — Full saved profile (same structure as GET /api/v1/profile/): basic_details, photos, religion_details, personal_details, location_details, family_details, education_details, about_me
  (Refresh token may also be set in HTTP-only cookie when JWT_REFRESH_COOKIE_NAME is set.)
  Error 400: Invalid OTP, user not found, or account not active.

2. PROFILE (/api/v1/profile/)
================================================================================
All profile endpoints require JWT authentication. Users can only access their own profile.
Error 401: Unauthorized if token missing or invalid.

Quick reference:
  GET    /profile/              Full profile (all sections)
  GET    /profile/basic/        Basic details only
  PATCH  /profile/basic/        Update basic details
  GET    /profile/photos/       Photos only
  PATCH  /profile/photos/       Upload/update photos (multipart)
  GET    /profile/religion/     Religion section
  PATCH  /profile/religion/     Update religion
  GET    /profile/partner-preferences/   Partner religion/caste preference
  PATCH  /profile/partner-preferences/  Update partner preference
  GET    /profile/personal/     Personal section
  PATCH  /profile/personal/     Update personal
  GET    /profile/location/     Location section
  PATCH  /profile/location/     Update location
  GET    /profile/family/       Family section
  PATCH  /profile/family/       Update family
  GET    /profile/education/    Education & occupation
  PATCH  /profile/education/   Update education
  GET    /profile/about/        About me text
  PATCH  /profile/about/        Update about me
  GET    /profile/generate-about/  Generate about-me suggestions
  POST   /profile/complete/     Mark profile complete (legacy)

--- Profile edit: 7 section endpoints (PATCH) ---
Profile creation and editing use the same section endpoints. Each section has one PATCH endpoint;
all fields in the body are optional (partial update). Auth: JWT required for all.

  1. Basic details
     PATCH /api/v1/profile/basic/
     Body (JSON): { "name": "string", "gender": "male"|"female"|"other", "dob": "YYYY-MM-DD", "email": "string" }
     Response 200: { "success": true, "data": { "name", "gender", "dob", "email", "phone" } }

  2. Location
     PATCH /api/v1/profile/location/
     Body (JSON): { "country_id": int, "state_id": int, "district_id": int, "city_id": int, "address": string }
     IDs must exist in master. Response 200: { "success": true, "data": { ... } }
     Error 400: Invalid country_id, state_id, district_id, or city_id.

  3. Religion
     PATCH /api/v1/profile/religion/
     Body (JSON): { "religion_id": int, "caste_id": int, "mother_tongue_id": int, "partner_religion_preference": string, "partner_preference_type", "partner_religion_ids", "partner_caste_preference" }
     Response 200: { "success": true, "data": { ... } }
     Partner preference also has dedicated GET/PATCH at /api/v1/profile/partner-preferences/

  4. Personal
     PATCH /api/v1/profile/personal/
     Body (JSON): { "marital_status_id": int, "children_count": int (≥0), "height_cm": int, "weight_kg": decimal, "colour": string, "blood_group": string }
     height_cm stored as free text (e.g. "168 cm"). Response 200: { "success": true, "data": { ... } }
     Error 400: Invalid marital_status_id.

  5. Education & occupation
     PATCH /api/v1/profile/education/
     Body (JSON): { "highest_education_id": int, "education_subject_id": int, "employment_status": string, "occupation_id": int, "annual_income_id": int }
     IDs must exist in master. Response 200: { "success": true, "data": { ... } }
     Error 400: Invalid id for any master reference.

  6. About me
     PATCH /api/v1/profile/about/
     Body (JSON): { "about_me": "string" }
     Response 200: { "success": true, "data": { "about_me": "..." } }

  7. Photos
     PATCH /api/v1/profile/photos/
     Content-Type: multipart/form-data
     Body (form): profile_photo, full_photo, selfie_photo, family_photo, aadhaar_front, aadhaar_back (each optional file)
     Response 200: { "success": true, "data": { "profile_photo", "full_photo", "selfie_photo", "family_photo", "aadhaar_front", "aadhaar_back" } }

(An additional section, Family, has PATCH /api/v1/profile/family/ with body: father_name, father_occupation, mother_name, mother_occupation, brothers, married_brothers, sisters, married_sisters, about_family — not part of the 7-step completion.)

--- 3a. Partner Preferences (religion/caste for matches) ---
GET /api/v1/profile/partner-preferences/
  Auth: Required (JWT)
  Returns structured partner preference used to filter matches. If user has no religion record, returns defaults.
  Response 200: { "success": true, "data": { "partner_preference_type", "partner_religion_ids", "partner_caste_preference" } }

  Auth: Required (JWT)
  Body (JSON): { "partner_preference_type": "own_religion_only"|"open_to_all"|"specific_religions", "partner_religion_ids": [int, ...], "partner_caste_preference": "any"|"own_caste_only" }
  All fields optional (partial update). When partner_preference_type is "specific_religions", partner_religion_ids must contain valid active religion IDs.
  Response 200: { "success": true, "data": { "partner_preference_type", "partner_religion_ids", "partner_caste_preference" } }

POST /api/v1/profile/personal/
  Auth: Required (JWT)
  Body (JSON):
    {
      "marital_status": "string (optional, one of master_marital_status names e.g. Never Married, Separated, Widowed, Divorced; matched by name in master, case-insensitive)",
      "has_children": "boolean (optional, required when marital_status is Separated, Widowed or Divorced)",
      "number_of_children": "integer (optional, required when has_children is true, ≥ 0)",
      "height": "integer (optional, height in cm, e.g. 175)",
      "weight": "decimal (optional, e.g. 65.5)",
      "complexion": "string (optional, one of: Fair, Wheatish, Dark, Very Fair)"
    }
  Validation rules:
    marital_status:
      - Optional. When provided, must match an active record in master_marital_status by name (case-insensitive).
      - If marital_status is "Separated", "Widowed" or "Divorced", has_children becomes required.
    has_children:
      - Optional normally.
      - When marital_status is "Separated", "Widowed" or "Divorced" and has_children is missing or null → 400 with message "Please specify whether you have children."
    number_of_children:
      - Required when has_children is true.
      - Must be an integer ≥ 0.
    height:
      - Optional integer in centimeters; stored as height in cm for the profile (e.g. 175).
    weight:
      - Optional decimal value (e.g. 65.0, 65.5).
    complexion:
      - Optional.
      - Must be one of: "Fair", "Wheatish", "Dark", "Very Fair" (case-sensitive).

  Example valid request:
    {
      "marital_status": "Separated",
      "has_children": true,
      "number_of_children": 2,
      "height": 175,
      "weight": 65.5,
      "complexion": "Wheatish"
    }

  Example invalid request (missing has_children for Separated):
    {
      "marital_status": "Separated"
    }

  Example invalid response 400:
    {
      "success": false,
      "error": {
        "code": 400,
        "message": "Please specify whether you have children."
      }
    }

  Success response 200:
    {

--- User Plan Details (My Plan) ---
  See section 3b "MY PLAN DETAILS" for full request/response. Response includes profile_views_remaining, interests_remaining, chat_remaining, contact_view_remaining, horoscope_remaining.
  Error 403 (when plan required): { "success": false, "error": { "code": 403, "message": "Your plan expired" } }

--- Plan limit logic ---
  PlanLimitService: can_view_profile, can_send_interest, can_chat, can_view_contact. Each action decrements the corresponding counter (profile_views_used, interests_used, chat_used, contact_views_used).
  View full profile → decrement profile_views_used; create ProfileView record.
  Send interest → decrement interests_used.
  Chat (initiation) → decrement chat_used (call when user starts chat; not done by permission API).
  View contact → decrement contact_views_used.

WebSocket — Real-time chat (Django Channels)
--------------------------------------------------------------------------------
  URL: ws://<host>/ws/chat/<conversation_id>/
  Authentication: JWT passed as query parameter: ?token=<access_token>
  Example: ws://127.0.0.1:8000/ws/chat/1/?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  The user must be a participant in the conversation; otherwise the connection is rejected (403).

  After connecting, send JSON text frames to post a message:
    { "message": "Hello!" }
  or
    { "text": "Hello!" }

  Incoming messages (broadcast to both participants) have the form:
    {
      "message_id": 101,
      "sender_id": "uuid",
      "sender_matri_id": "AM100001",
      "sender_name": "Anna",
      "text": "Hello!",
      "created_at": "2026-03-13T20:55:00Z"
    }

  Messages are persisted to the database and conversation.updated_at is updated on each new message.
  Backend uses Redis channel layer (channels_redis) for broadcasting between workers.

================================================================================
4a. WISHLIST APIs (/api/v1/wishlist/)
================================================================================
All endpoints require: Authorization: Bearer <access_token>

Wishlist stores profiles that the logged-in user has favorited (heart icon on matches page).

--- TOGGLE WISHLIST ---
POST /api/v1/wishlist/toggle/
  Auth: Required (JWT)
  Body (JSON):
    { "matri_id": "AM100012" }
  Logic:
    - Find target profile by matri_id.
    - If user tries to wishlist their own profile → 403.
    - If profile already in wishlist → remove it.
    - Otherwise add it.
  Success 200 (added):
    {
      "success": true,
      "data": { "is_wishlisted": true }
    }
  Success 200 (removed):
    {
      "success": true,
      "data": { "is_wishlisted": false }
    }
  Error 400: matri_id missing.
  Error 403: Cannot wishlist own profile.
  Error 404: Profile not found.

--- LIST USER WISHLIST ---
GET /api/v1/wishlist/
  Auth: Required (JWT)
  Query params:
    page  (default 1)
    limit (default 10, max 50)
  Response 200:
    {
      "success": true,
      "data": {
        "total": 3,
        "page": 1,
        "limit": 10,
        "profiles": [
          {
            "matri_id": "AM100012",
            "name": "Priya Sharma",
            "age": 26,
            "location": "Mumbai, Maharashtra",
            "education": "MBA",
            "occupation": "Software Engineer",
            "profile_photo": "https://cdn.site/profile/12.jpg",
            "match_percentage": 95,
            "is_online": true,
            "last_seen": "10 minutes ago"
          }
        ]
      }
    }

--- REMOVE FROM WISHLIST ---
POST /api/v1/wishlist/remove/
  Auth: Required (JWT)
  Body (JSON):
    { "matri_id": "AM100012" }
  Logic:
    - If profile exists in wishlist → remove it.
    - If not present, operation is idempotent (still returns success).
  Success 200:
    { "success": true, "message": "Profile removed from wishlist." }
  Error 400: matri_id missing.
  Error 404: Profile not found.

================================================================================
ERROR RESPONSE FORMAT (DRF custom handler)
================================================================================
All error responses:
{
  "success": false,
  "error": {
    "code": 400 | 401 | 403 | 404 | 429 | 500,
    "message": "Human-readable message",
    "details": { ... validation or extra detail ... }
  }
}
  400: Bad request (validation, invalid IDs, etc.)
  401: Unauthorized (missing or invalid JWT)
  403: Forbidden (e.g. admin-only action)
  404: Not found
  429: Too many requests (e.g. OTP rate limit)
  500: Server error
